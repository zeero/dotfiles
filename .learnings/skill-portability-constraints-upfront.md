### skill の可搬性は設計・実装の工程に観点として組み込む

**学び**: skill の可搬性要件（project-agnostic・ハードコードパスなし・他 skill への依存を安易に足さない・user-invoke-only 既定）は、事後に発見して直すものではなく、設計時に要件として明示し、実装時・提示前にチェック項目として通す。担保の手段（セルフチェック、スクリプト、監査エージェント等）は状況に応じて選べばよく、本質は「可搬性の観点を工程に前置きすること」にある。

**根拠**: insights 分析（2026-07）で、skill 設計が特定プロジェクトへ過剰結合し毎回2ラウンドの事後修正が発生していた。観点が工程に組み込まれておらず、提示後にユーザ指摘で発覚するのが原因だった（2026-07-06）。

**適用範囲**: skill の新規作成・改修全般（skill-creator / loop-skill-creator 経由を含む）。

**例外・前提**: 意図的にプロジェクト固有へ寄せる skill（repo 専用の手順書など）はこの限りではない。チェック項目が決定論的に検証できるほど固まったら script 化を検討する（skill-deterministic-to-script の原則）。
