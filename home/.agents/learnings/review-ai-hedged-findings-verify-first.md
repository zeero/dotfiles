# レビューAIの「保留付き指摘」は実コード検証してから採否を決める

## 状況

Codex に計画/issue本文をレビューさせたところ、ある指摘を **Critical** 重大度で出してきた。
内容は「本文が断定している関数名・protocol メソッド名（例: `count_active_positions()`、`list_active_positions()`）は実コードに存在しないかもしれないので、一般表現に薄めるべき」というもの。
ただし指摘自体に「逐語名を要確認」という保留が付いていた。

## 結論

**レビューAIが「要確認 / 〜かもしれない」と保留を付けた指摘は、実コードを grep/Read で逐語照合してから採否を決める。** 保留付き指摘を鵜呑みにして従うと、改悪になることがある。

このケースでは照合した結果、断定されていた関数名は **すべて実在し正確** だった（`db.py` 定義 → `entry_flow.py` で実使用、`ports.py` の protocol メソッドも逐語一致）。
Codex 案どおり「正確な関数名を曖昧な一般表現に薄める」修正をしていたら、**着手時に参照すべき正しいシンボル名をわざわざ消して逆効果** になっていた。

レビューAIは「ファイルを読めば確定できる事実」でも、コンテキストに無いと安全側に倒して "要確認" 付きで指摘を盛る傾向がある。重大度ラベル（Critical 等）も額面どおりには受け取らない。

## 適用の決め手

- 指摘文に保留語（要確認 / おそらく / かもしれない / 未確認）があるか → あれば検証フェーズを必ず挟む
- 指摘が「具体的なシンボル名・ファイル名・行」に関するものか → grep/Read で即検証できる。AIの推測より一次情報を優先
- 「曖昧化・一般化せよ」という方向の修正提案は特に警戒 → 正確な情報を失う改悪になりやすい

受け取ったレビューに機械的に従わず検証する姿勢は `superpowers:receiving-code-review` の精神とも一致する。
