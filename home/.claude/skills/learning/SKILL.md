# Learning

作業で得た「次回から楽になる学び」を記録し、後で再利用できるようにするスキル。

**なぜやるか**: ここに書いた学びは、次に同じ作業へ出会ったとき（自分や他のエージェントが）参照され、調べ直しややり直しの時間を減らす。だから「次に時間を節約できる知見」を選んで書く。

このスキルが属する複利エンジニアリング全体（2つの軸・置き場・サイクル・各スキルの役割）の地図は [README.md](./README.md) を参照。

ユーザからの補足（あれば考慮する）: $ARGUMENTS

## プロセス

### 0. セットアップ

複利エンジニアリングの置き場を用意する。**毎回このステップを通る**。

まず存在ゲートと決定論的セットアップをスクリプトに任せる:

```bash
~/.claude/skills/learning/scripts/setup.sh
```

スクリプトはマシン共通（`~/.agents/{learnings,reflection}/`）と repo-local（`<repo>/LEARNINGS.md`・`STRATEGY_SPEC.md`）の有無を各々独立に判定し、不足分だけをテンプレからコピーする（既存は上書きしない・冪等）。CLAUDE.md には触れない。

出力が `all set` なら、この節はここで終わり。`ACTION NEEDED (model)` が出たら、**その項目だけ**を次の要領で対応する（出るのは今回作成した分のみ）。

- **`[global]` compound-rules 追記** — グローバル context file（`~/.claude/CLAUDE.md` 等）に `assets/templates/global/compound-rules.md` の方針節を追記する。**グローバル設定の編集なので必ずユーザに確認してから行う**。
- **`[repo]` STRATEGY_SPEC.md の埋め込み** — スクリプトが置いた素の `STRATEGY_SPEC.md` を、**repo の README / CLAUDE.md / docs から自明に分かる範囲（目的・方針・指標）で埋める**。丸投げしない。人の判断が要る箇所だけ `TODO:` で残し、埋めた内容と残した TODO をユーザに報告する。
- **`[repo]` 作業サイクル節の追記** — repo の CLAUDE.md に「作業サイクル」節が無ければ `assets/templates/project/cycle-section.md` の内容を追記する。**既存 CLAUDE.md の編集はユーザに確認してから行う**。
  - 追記時、その repo で使える review / refactor / test 系 skill（例: `self-code-review`, `tdd-refactoring`）を該当ステップに紐づけ、セッション締めの `retrospective`・判断の見返しの `reflect` を末尾に添える。該当 skill が無ければ汎用テンプレのまま残す。
  - review ステップの「後戻りしにくい・影響範囲が広い・迷った判断」という汎用トリガは、repo 固有の重要判断の例（戦略の設計選択、構成・可搬性に関わる選択など）に置き換えて具体化する。STRATEGY_SPEC と同じく自明な範囲まで埋め、判断材料が無ければ汎用のまま残す。

### 1. 学びを1つ取り出す

今回の作業から、別の場面でも再利用できる知見を抽出する。1件ずつ、簡潔に。

### 2. 保存先を決める

| 学びの種類 | 保存先 |
| :--- | :--- |
| そのリポジトリ固有 | `<repo>/LEARNINGS.md` |
| プロジェクトをまたいで使える | `~/.agents/learnings/` |

既定は repo 固有の `LEARNINGS.md`。共有ストア（グローバル領域 `~/.agents/learnings/`）に置くと判断したら、**新規・移動を問わず、書き込み前に内容と保存先をユーザに提示して承認を得てから反映する**（グローバル設定の編集と同じ扱い）。迷ったら repo 固有に置く。

### 3. <learning-entry-format> の書式で書く

<learning-entry-format>
```markdown
### タイトル

**学び**: 何が効くか / 何を避けるか

**根拠**: いつ・どこで分かったか

**適用範囲**: どんな場面で使えるか

**例外・前提**: 使えない場面・前提条件
```
</learning-entry-format>

### 4. よく使う学びは共有ストアへ移す

`LEARNINGS.md` の学びが、このリポジトリの外でも使えると判断でき、「適用範囲」と「例外」が書ける状態になったら、`~/.agents/learnings/` へ移す。

- 移すのもグローバル領域への書き込みなので、step 2 の承認ゲートに従う（内容と保存先を提示し承認を得てから反映）。
- 本体は個別ファイル（`~/.agents/learnings/<topic>.md`）に書き、`~/.agents/learnings/index.md` には `タイトル — ファイル名 — 1行要約` を1行だけ追記する（index は短く保つ）。
- 移したら元の `LEARNINGS.md` のエントリは削除する（コピーではなく移動）。可視性はグローバル `index.md`（毎セッション読み込み）で保たれるので、repo-local に残すと二重管理になる。

### 5. セッションを振り返り、記録漏れの判断がないか確認する

学びの記録を終えたら、最後に review ステップで `decisions-log.md` に記録すべきだった判断の取りこぼしがないか振り返る。**判断基準は review ステップ（作業サイクル節）に従い、ここでは繰り返さない**。漏れていれば「選択・理由・代替案・確信度」を補記し、無ければ何も書かない（無理に捻り出さない）。昇格（→ `patterns.md`）は `reflect` の仕事なのでここではやらない。チェック結果（補記の有無）は次の報告にまとめて伝える。

### 6. 報告する

今回 記録した学び（共有ストアへ移したものがあればそれも）を一覧でユーザに報告する。step 5 の判断チェックの結果（補記したものがあればそれも、無ければ「記録漏れなし」）も併せて伝える。

## 注意

- すでに記録済みの内容は重複させない。
- 似た学びが2つあれば1つに統合する。
- 後で誤りと分かった学びは消さず、「撤回: 理由」を追記して残す（同じ失敗を繰り返さないため）。
- 一度に詰め込まず、数件までに絞る。
