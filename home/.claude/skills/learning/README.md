# 複利エンジニアリング — 全体像

作業のたびに「再利用できる資産」を貯め、次のサイクルを毎回少しずつ楽に・確実にしていく仕組み。`learning` スキルはこの仕組みの記録係であり、置き場の bootstrap も兼ねる。

## 1つの軸 — 汎用性

貯める知見はすべて「学び」であり、割り方は知見の**性質**ではなく**汎用性（どこまで通用するか）**の1軸だけ。どれだけ汎用的に使えるかで置き場が決まる。

| 汎用性 | 置き場 | ロード |
| :--- | :--- | :--- |
| あらゆる作業で効く普遍的な話 | グローバル CLAUDE.md（実体 `~/.claude/CLAUDE.md`） | 常時 |
| 特定領域における普遍的な話 | skill | オンデマンド（description で trigger） |
| プロジェクト固有だが再利用性のある話 | `<repo>/.learnings/`（`index.md` ＋ `<slug>.md`） | index を常時、本体は辿る |

## 思想（なぜ汎用性の1軸か）

- **性質で割らない**。かつては「学び（何が効くか）」と「省察（どう判断したか）」を別レベルとして2軸で管理していたが、省察の産物（判断の型）も突き詰めれば「普遍的な話」であり、置き場は汎用性で決まる。性質での区別は分類コストを生むだけなので畳んだ。
- **蒸留が唯一のレバー**。学びは捕捉した時点で「どれだけ汎用的に使えるか」を見極め、正しい置き場へ置く。ここを丁寧にやることがすべて。普遍な型はグローバル CLAUDE.md に置かれ常時ロードされるので、「判断の前に必ず読む」といった能動的なゲートを別に用意しなくても、自然に視界へ入って効く。
- **後追い昇格の機構は作らない**。「後から汎用性が判明する」のは捕捉時の抽象化を誤ったサインであり、機構で補償するのは一時しのぎ。抽象化を捕捉時に正す。（メモリストアからの回収は後追い昇格ではない — learning を通らずに書かれたメモへ、捕捉時蒸留を初めて適用する入口である）
- **再登場は昇格の合図ではなく、ループの故障**。同じ学びに再遭遇するのは、置いた学びが使用時点に surface していない（効いていない）ということ。昇格で報いず、なぜ効かなかったか（配置・description・index 要約）を根本原因として直す。
- **なぜ「複利」か**。正しい層に置かれた学びが以後の全作業の入力になり、積むほど次が速く・確実になる。

## 構成要素

repo-local（そのリポジトリに置く）と横断（マシン共通）の2層。

```
repo-local（そのリポジトリに置く）
<repo>/
├── .learnings/
│   ├── index.md                — 1行要約の索引。@import で常時ロードされる入口
│   └── <slug>.md               — 学び本体（1トピック1ファイル、index から辿る）
└── CLAUDE.md ＞「作業サイクル」節 — サイクルの回し方＋ @.learnings/index.md で index を常時ロード

横断（マシン共通）
~/.claude/CLAUDE.md（/ AGENTS.md）
└── コア原則・各種方針           — 「あらゆる作業で効く普遍的な話」の置き場そのもの
skill（~/.claude/skills/）
└── <domain-skill>              — 「特定領域における普遍的な話」の置き場
```

横断の学びは専用のストアを持たず、グローバル CLAUDE.md 本体か skill に溶かす。個別ファイルの横断ストアは持たない（汎用性の高い知見は、常時ロードの原則か、trigger で引ける skill として直接効かせるのが筋だから）。

## コンテキストの繋がり方

```
毎セッション読み込み
  グローバル CLAUDE.md（~/.claude/CLAUDE.md / AGENTS.md）
    └ 普遍的な学びは本文に溶けているので、そのまま常に効く

repo で作業中はさらに読み込み
  <repo>/CLAUDE.md ＞ 作業サイクル節
    └ @.learnings/index.md … index（1行要約）が毎回視界に入り、必要な <slug>.md だけ辿る
```

ポイントは2つ。

- **常時ロードは軽く保つ** — プロジェクトの学びは `index.md`（1行要約だけ）を入口にして、本体 `<slug>.md` は必要なときだけ辿る。
- **`@import` はツール依存** — `@.learnings/index.md` は Claude Code でのみ自動ロードされる。Codex / Pi では素のポインタ（リンク）として扱われ、エージェントが必要時に開く。なお `@path` を backtick で囲むと展開されないので、自動ロードさせたいパスは囲まない。

## スキルの役割

- **`learning`** — 作業の区切りやセッションの締めで学びを記録し、汎用性を見極めて正しい層へ蒸留する。セッション全体を振り返って汎用知見を抽出する抽象化フェーズ（旧 `retrospective` 由来）も担う。置き場の bootstrap 入口も兼ねる。
- **`learning-from-memory`** — メモリストア（auto-memory / Serena）に learning を通らず溜まった未蒸留の汎用知見を回収し、learning の置き場へ昇格させて元エントリを後処理する（明示依頼時のみ起動）。蒸留の判定基準と承認ゲートは learning のプロセスを再利用する。

## 蒸留（昇格）の流れ

捕捉した学びは、その場で汎用性を見極めて置き場を決める（[SKILL.md](./SKILL.md) のプロセス2）。

- 普遍的 → グローバル CLAUDE.md（要ユーザ承認）
- 特定領域で普遍 → skill（新規作成はサブエージェントへ委譲。要ユーザ承認）
- プロジェクト固有 → `<repo>/.learnings/`

## メモリストアとの関係

Claude Code の auto-memory や Serena memory のような「セッション中に低摩擦で自由書き込みされるストア」は、この仕組みの一次捕捉層（inbox）として扱う。書き込みは制約しない。learning を通らずに溜まった汎用知見は `learning-from-memory` が回収・蒸留し、昇格後は元エントリを削除（またはストア固有の残滓だけに書き換え）して二重管理を防ぐ。エージェント固有の振る舞い調整・ユーザ個人の好み・進行中の文脈・ポインタは各ストアの本来の役割なので、対象外として留まる。

依存は learning 側 → ストアの一方向のみ。ストア側（auto-memory の書き込みや dreaming-auto-memory などの整理機構）は learning を知らず、単独で成立する。
