# Takeover - セッション情報の復元

`.claude/HANDOVER.md` から前回のセッション情報を読み込み、状況を把握して次のアクションを提案します。

## 実行手順

### 1. ハンドオーバーファイルの確認

- `.claude/HANDOVER.md` が存在するか確認
- ファイルが存在しない場合:
  - ユーザーに「HANDOVER.md が見つかりません」と報告
  - 新規セッションとして開始するか、`/handover` で作成するか提案
  - 処理を終了

### 2. ファイル読み込み

- Read ツールを使用して `.claude/HANDOVER.md` の全内容を読み込む

### 3. コンテキストの復元

ハンドオーバードキュメントを詳細に分析し、以下の順序でコンテキストを再構築する:

#### 3-1. 背景とコンテキストの理解（最優先）

- 前回セッションが取り組んでいた課題・目標の背景を把握する
- 作業対象のコードベース・モジュール・機能領域を理解する
- 前提条件や制約（環境依存、バージョン制約、チーム合意など）を把握する

#### 3-2. 参照情報の確認

- HANDOVER.md に記載された参照ドキュメント・URL を確認する
- 特に重要な参照情報は、必要に応じて Read や WebFetch で最新の状態を再取得する
- 暗黙知として記録された情報（ドキュメントに書かれていないが重要な知見）を把握する

#### 3-3. 作業状況の把握

- 達成された成果と未完了のタスク
- 重要な技術的決定と背景
- 遭遇した問題と解決策
- 学んだ教訓と注意点
- 重要なファイルとその役割

### 4. 現状との差分確認

- 前回のセッション以降にファイルが変更されていないか、git status / git log で確認する
- 重要なファイルについて Read して最新の状態を把握する
- HANDOVER.md の記述と現状に乖離がある場合、ユーザーに報告する

### 5. 状況報告

ユーザーに以下の形式で状況を報告:

#### 📋 前回セッションのサマリー

- 背景と目標
- 取り組んでいたタスク
- 達成されたこと

#### 📚 引き継がれたコンテキスト

- 参照ドキュメント・URL のリスト
- 暗黙知・重要な知見
- 前提条件と制約

#### ⚠️ 注意が必要な事項

- 未解決の問題
- 既知の制限事項
- 注意すべき gotchas
- 前回セッション以降の変更（検出された場合）

#### 💡 前回の重要な決定

- 技術選択とその理由
- アーキテクチャ判断
- トレードオフ

#### 🗺️ 重要なファイル

- 前回変更されたファイル
- 今後変更が必要なファイル
- キーとなるコードの場所

#### 🎯 提案された次のステップ

- 前回提案された次のアクション
- 優先順位順にリスト化
- 各アクションの依存関係

### 6. 次のアクション提案

前回の HANDOVER.md の内容に基づき、具体的な次のアクションを提案:

- 続きから作業を再開する選択肢
- 前回の問題を解決する選択肢
- 新しい方向性を探る選択肢
- ユーザーに「どこから始めますか？」と尋ねる

アルファベット番号付き箇条書きで提示（a, b, c, d など）

## 重要な注意事項

- HANDOVER.md の内容を正確に理解する
- **背景・コンテキスト・参照情報を最優先で復元する**（作業内容より先に文脈を理解する）
- 前回の Claude の意図や文脈を尊重する
- ユーザーに状況を明確に伝え、次のアクションを選びやすくする
- 前回からの継続性を保ちつつ、新しい視点も提供する
- 必要に応じて、前回の決定や方針について再確認する
- 参照情報が古くなっている可能性を考慮し、重要なものは再取得する
