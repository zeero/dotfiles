# Handover - セッション引き継ぎドキュメント生成

このセッションでの作業内容を振り返り、次のセッションへの引き継ぎドキュメントを `.claude/HANDOVER.md` に生成します。

## 引数

- `[focus-area]` (オプション): 特に詳細に記録したい内容やトピックを指定できます
  - 例: `/handover "API統合の実装詳細"`
  - 例: `/handover "バグ修正の経緯と学び"`

## 実行手順

### 1. 既存ファイルのクリーンアップ

- `.claude/` ディレクトリが存在するか確認し、存在しない場合は作成: `mkdir -p .claude`
- `.claude/HANDOVER.md` が既に存在するか **Bash で `test -f` を使って確認**する
  - **⚠️ 重要: 既存の HANDOVER.md を Read ツールで読み込んではならない**（前回の記述に引きずられ、今回のセッション固有の情報が薄まるため）
  - 存在する場合は、Bash で `rm` して削除する

### 2. セッション内容の分析

このセッション全体を振り返り、以下の観点で情報を整理する:

#### 必須セクション

1. **📋 背景とコンテキスト**
   - このセッションで取り組んだ課題・目標の背景
   - 作業対象のコードベース・モジュール・機能領域の説明
   - 前提条件や制約（環境依存、バージョン制約、チーム合意など）
   - 次のセッションが「なぜこの作業をしているのか」を理解するのに十分な情報

2. **📚 参照情報**
   - セッション中に参照した公式ドキュメント・リファレンスのURL
   - 参照した Issue、PR、Slack スレッドなどのリンク
   - 調査で得た知見のうち、ドキュメントに書かれていない重要な情報
   - 参照した CLAUDE.md、steering、spec などの内部ドキュメント
   - 各参照の要点を1行で添えること（リンク切れ時にも意味が通るように）

3. **📊 作業概要**
   - このセッションで取り組んだ主要なタスク
   - 達成された成果物
   - 変更されたファイルのリスト

4. **✅ うまくいったこと**
   - 成功した実装やアプローチ
   - 効果的だった手法やツール

5. **❌ うまくいかなかったこと**
   - 遭遇したバグとその原因
   - 失敗したアプローチとその理由
   - 修正方法と解決策

6. **💡 主要な決定**
   - アーキテクチャやデザインの決定
   - 技術選択とその理由
   - トレードオフと選択の根拠

7. **⚠️ 学んだ教訓と注意点**
   - 発見された gotchas や落とし穴
   - 将来同じ問題を避けるための知見
   - 暗黙知として残すべき情報（コードやドキュメントからは読み取れないもの）

8. **🎯 次のステップ**
   - 明確で具体的な次のアクション
   - 優先順位付き
   - 各ステップの依存関係や前提条件

9. **🗺️ 重要なファイルマップ**
   - 変更されたファイルとその役割
   - 新規作成されたファイル
   - 今後変更が必要になりそうなファイル
   - 各ファイルの簡単な説明

#### Focus Area (引数が指定された場合)

$ARGUMENTS

- 上記については、特に詳細なセクションを追加
- 以下の情報を含める:
  - 該当トピックの詳細な実装内容
  - 技術的な詳細や設計判断
  - コードスニペットや設定例
  - 関連するファイルパス

### 3. ドキュメント生成

- 上記の内容を構造化されたMarkdown形式で整理
- 見出しに絵文字を使用して視認性を向上
- コードブロックやリストを適切に使用
- 簡潔だが十分な情報量を保つ（標準：バランス重視）
- **「次のセッションのAIがゼロから読んでも作業を再開できる」ことを品質基準とする**

### 4. ファイル保存

- Write ツールを使用して `.claude/HANDOVER.md` に保存

### 5. 完了報告

- ユーザーに以下を報告:
  - ✅ HANDOVER.md が生成されたこと
  - 📂 ファイルパス: `.claude/HANDOVER.md`
  - 📝 含まれるセクションの概要
  - 次回セッションで `/takeover` を使用して情報を復元できること

## 重要な注意事項

- **既存の HANDOVER.md は絶対に読み込まない**（存在確認のみ許可）
- セッション全体を包括的に振り返る
- 技術的な詳細と文脈の両方を含める
- 次の Claude が引き継ぎやすいよう、明確で具体的に記述
- 主観的な評価ではなく、事実ベースで記録
- コードや設定の変更は具体的なファイルパスと共に記録
- 暗黙知（コードを読んだだけでは分からない情報）を意識的に言語化する
